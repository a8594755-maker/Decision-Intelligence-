# Risk Dashboard 快速入門指南

## 🚀 快速開始

### 1. 啟動應用

```bash
npm run dev
```

### 2. 訪問 Risk Dashboard

1. 登入應用
2. 點擊左側選單 **Planning** → **Risk Dashboard**

---

## 📊 功能說明

### 紅綠燈風險儀表板

**目的**: 快速識別有斷料風險的零件

**視覺化**:
- 🔴 **Critical** (< 7 天): 紅色背景，需立即處理
- 🟡 **Warning** (7-14 天): 黃色背景，需密切關注
- 🟢 **Low** (≥ 14 天): 正常，庫存充足

### KPI 卡片

顯示四個關鍵指標：
1. Critical 風險數量
2. Warning 風險數量
3. Low 風險數量
4. 總料號數

### 風險表格

| 欄位 | 說明 |
|-----|------|
| 料號 | 零件料號 |
| 工廠 | 工廠代碼 |
| 現有庫存 | 當前庫存數量 |
| 日均消耗 | 每天平均消耗量 |
| 撐幾天 | 距離斷料天數 |
| 斷料機率 | 斷料的可能性 (%) |
| 風險等級 | Critical/Warning/Low |
| 操作 | 查看詳細資訊 |

### 篩選功能

1. **工廠篩選**: 只顯示特定工廠的資料
2. **風險等級篩選**: 
   - 全部等級
   - 只顯示 Critical
   - 只顯示 Warning
   - 只顯示 Low

## 🧮 計算邏輯

### 1. 距離斷料天數

**公式**: `(現有庫存 - 安全庫存) ÷ 日均消耗`

**範例**:
```
現有庫存: 100
安全庫存: 20
日均消耗: 10

距離斷料天數 = (100 - 20) ÷ 10 = 8 天
```

### 2. 斷料機率

根據庫存天數與提前期的比例：

| 條件 | 機率 |
|-----|------|
| 庫存 < 提前期 × 0.5 | 90% |
| 庫存 < 提前期 | 70% |
| 庫存 < 提前期 × 1.5 | 30% |
| 其他 | 10% |

**波動調整**: 需求波動 > 20% 時，機率 +10%

### 3. 緊迫分數

| 距離斷料天數 | 分數 | 等級 |
|-------------|------|------|
| < 7 天 | 100 | Critical |
| 7-14 天 | 50 | Warning |
| ≥ 14 天 | 10 | Low |

## 🎯 使用場景

### 場景 1: 每日晨會

**目標**: 快速識別當天需處理的緊急料號

**步驟**:
1. 開啟 Risk Dashboard
2. 篩選器選擇「只顯示 Critical」
3. 按緊迫分數排序（預設已排序）
4. 處理前 10 個最危險的料號

**預期結果**: 5 分鐘內完成風險檢視

### 場景 2: 工廠別風險監控

**目標**: 檢視特定工廠的庫存狀況

**步驟**:
1. 篩選器選擇目標工廠（如 P001）
2. 檢視該工廠的 Critical 和 Warning 數量
3. 點擊料號查看詳細資訊

**預期結果**: 了解該工廠的整體風險狀況

### 場景 3: 採購計畫

**目標**: 根據風險優先級安排採購

**步驟**:
1. 檢視所有 Critical 料號
2. 點擊料號查看詳細資訊（提前期、斷料機率）
3. 根據緊迫分數決定採購順序

**預期結果**: 建立優先順序清單

## ⚠️ 空狀態處理

### 情況 1: 無庫存資料

**顯示訊息**:
```
尚無庫存資料，請先至資料上傳頁面匯入 Inventory Snapshot
```

**解決方法**:
1. 前往 **Data** → **Data Upload**
2. 上傳 `inventory_snapshots.csv` 或 `.xlsx` 檔案
3. 返回 Risk Dashboard 重新整理

### 情況 2: 資料表不存在

**錯誤訊息**:
```
尚未建立 inventory_snapshots 表，請聯絡管理員
```

**解決方法**: 請管理員執行資料庫 Schema 建立

## 📝 資料格式要求

### inventory_snapshots 表

| 欄位 | 型別 | 必填 | 說明 |
|-----|------|------|------|
| material_code | text | ✅ | 料號 |
| plant_id | text | ✅ | 工廠代碼 |
| on_hand_qty | numeric | ✅ | 現有庫存 |
| safety_stock | numeric | ✅ | 安全庫存 |
| lead_time_days | numeric | ✅ | 提前期（天） |
| demand_volatility | numeric | ❌ | 需求波動（預設 0.1） |

### component_demand 表

Risk Dashboard 會從 `component_demand` 表計算日均需求。

**計算方式**:
- 取該料號的所有未來需求
- 計算平均週需求
- 除以 7 得到日均需求

## 🧪 測試範例

### 測試案例 1: Critical 風險

**模擬資料**:
```javascript
{
  material_code: 'COMP-001',
  plant_id: 'P001',
  on_hand_qty: 50,
  safety_stock: 20,
  lead_time_days: 10,
  demand_volatility: 0.15,
  // 假設日均需求 = 10
}
```

**預期結果**:
- 距離斷料: 3 天 `(50-20)/10`
- 斷料機率: 90% `(3 < 10*0.5)`
- 緊迫分數: 100 `(3 < 7)`
- 風險等級: 🔴 Critical

### 測試案例 2: Warning 風險

```javascript
{
  material_code: 'COMP-002',
  plant_id: 'P001',
  on_hand_qty: 100,
  safety_stock: 0,
  lead_time_days: 7,
  demand_volatility: 0.1,
  // 假設日均需求 = 10
}
```

**預期結果**:
- 距離斷料: 10 天 `(100-0)/10`
- 斷料機率: 70% `(10 < 7*1.5)`
- 緊迫分數: 50 `(7 ≤ 10 < 14)`
- 風險等級: 🟡 Warning

### 測試案例 3: Low 風險

## 🛠️ 故障排除

### 問題 1: 載入失敗

**症狀**: 一直顯示「載入中...」

**可能原因**:
1. 未登入
2. 資料表不存在
3. 資料庫連線問題

**解決方法**:
1. 確認已登入
2. 檢查瀏覽器 Console 錯誤訊息
3. 重新整理頁面

### 問題 2: 無資料顯示

**症狀**: 表格顯示「無符合條件的資料」

**可能原因**:
1. 尚未上傳庫存資料
2. 篩選器設定過嚴

**解決方法**:
1. 上傳庫存資料
2. 清除所有篩選器

### 問題 3: 計算結果異常

**症狀**: 斷料天數或機率看起來不對

**可能原因**:
1. 日均需求為 0
2. 安全庫存設定錯誤
3. 資料本身有誤

**解決方法**:
1. 點擊詳細資訊查看計算過程
2. 檢查原始資料是否正確
3. 確認 component_demand 資料存在

## 💡 最佳實踐

### 1. 每日檢視

建議每天早上第一件事就是檢視 Risk Dashboard：
- 重點關注 Critical 料號
- 追蹤昨天的 Warning 是否惡化
- 建立當日行動計畫

### 2. 設定篩選器

針對不同角色設定不同的預設篩選：
- **採購主管**: 只看 Critical
- **工廠經理**: 篩選自己的工廠
- **計畫人員**: 看全部資料

### 3. 定期更新資料

- 每天更新 inventory_snapshots
- 每週更新 component_demand
- 確保資料時效性

### 4. 結合其他工具

Risk Dashboard 可與其他頁面配合使用：
- **Forecasts**: 查看未來需求趨勢
- **BOM Data**: 了解零件結構
- **Supplier Management**: 聯絡供應商
